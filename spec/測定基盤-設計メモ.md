# 測定基盤（記録レベル）設計メモ（2026-07-16 着工）

親の要望：「どの子が・どの程度の時間で・どこまで進んで・どんな結果か・学年別の平均は」を
先々まで測れるようにする。3段階で作る（相談済み・第1〜2段階を着工）。

━━━━━━━━━━━━━━━━━━━━━━
## 前提（今ある記録と2つの穴）

- 1問ごとの明細（log：t時刻/d日付/h時間帯/sub教科/cat分野/ok正誤/ms回答時間/p何問目/u単元）と
  バトルごとのセッション（sessions：d/mins分数/count/sub/result）は記録済み＝材料は揃っている
- 穴1：log は直近3,000問・sessions は300件で古いものから消える（長期測定が不可能）
- 穴2：見守りは家族単位のみ。学年別平均は家族横断＝別の窓口が必要

## 第1段階：見守り画面の強化（admin.html のみ・DB変更なし）【今回実装】

- 各プレイヤーのカードに追加：
  - 学年バッジ（保存データの grade）
  - きょうの学習時間（分）／この7日間の学習時間（分）＝ sessions を日別集計
  - 1問あたりの平均回答時間（秒）＝ log の ms 平均
  - 7日間グラフの下に「合計◯問・◯分」の行
- データ源は既存の sq_admin_overview のまま（変更なし・閲覧専用の性質も維持）

## 第2段階：日別サマリーの倉庫（Supabase・穴1の解決）【SQL準備済み・適用は承認後】

- 新テーブル save_days（player_id×日付が主キー）：n問数・ok正解数・ms合計・mins分数・grade
- **sq_put_save を差し替え**：セーブのたびに、送られてきた JSON の log/sessions を
  日別に集計して save_days へ upsert →ゲーム側（index.html）は無改修で穴1が塞がる
  - log から消えかけの日（件数が減って見える日）は「過去最大の記録」を保持
    （ON CONFLICT の WHERE excluded.n >= save_days.n）
  - Studio の DO ブロックバグ回避のため、関数は「CTE を連ねた単文」で書く
- 家族向けの履歴窓口 sq_admin_days（あいことば＋ばんごう）を新設（第1段階の将来拡張用）
- 適用手順：sql/2026-07-16-save-days.sql を Supabase SQL Editor に貼って Run
  （本番の保存経路を触るため、適用はユーザーの明示承認後。適用後に実書き込みテスト）

## 第3段階：学年別平均ダッシュボード【未着工・利用家族が増えてから】

- save_days×players×families を学年で横断集計する管理人専用の窓口＋画面
- 個人情報保護：平均値だけを返す（他の家の子の個別データは出さない）
- 管理人の鍵は「あいことば」とは別系統にする（全家族を見られる鍵のため）
- 母数が「うちの子だけ」のうちは平均に意味が薄い→川口など配布が進んでから

## やらないこと

- ゲーム側（index.html）の変更（難易度底上げ・川口と衝突させない）
- 「診断」的な表現（見立ての方針を維持）
